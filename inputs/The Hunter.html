Here is the formal Product Requirement Document (PRD) for building "The Hunter" as an in-house asset.

This document is structured for a Technical Architect/Senior Developer to immediately begin component selection and coding.

📄 PRD: "The Hunter" (Internal Lead Gen Agent)

Version: 1.0 (MVP)

Type: In-House Proprietary Asset

Objective: To replace expensive SaaS subscriptions (Apollo/PhantomBuster) with a custom Python-based agent that autonomously identifies, enriches, and drafts outreach for local B2B leads.

1. Executive Summary

"The Hunter" is an autonomous Python script that runs daily. It scrapes local business data from Google Maps based on a niche keyword, visits their websites to extract contact details (Email/Phone) and "Context" (recent updates), and uses a low-cost LLM (DeepSeek/Groq) to draft hyper-personalized outreach messages.

Primary Goal: Generate 50 qualified, drafted leads per day for the sales team to review.

Cost Constraint: < $10/month operational cost.

3. Technical Architecture & Component Selection

Based on the "In-House Expert" strategy (Low Cost, High Control).

3.1 Core Components

  1. Input Source: config.json (Stores keywords like ["Dentists in Baner", "Gyms in Kothrud"]).
  2. Discovery Module (Scraper):
    • Technology: Python + Playwright (Faster/more reliable than Selenium).
    • Target: Google Maps (Web interface).
    • Risk Control: Headless browsing with User-Agent rotation.
  3. Enrichment Module (Crawler):
    • Technology: BeautifulSoup4 + Requests.
    • Logic: Timeout set to 10s. Regex search for email patterns (r"[a-z0-9\.\-+_]+@[a-z0-9\.\-+_]+\.[a-z]+") but exclude generic extensions like .png or .jpg.
  4. Intelligence Module (The Brain):
    • Model: DeepSeek-V3 (via API) or Llama-3-8B (via Groq Cloud).
    • Why: Ultra-low latency and 95% cheaper than GPT-4o for this specific task.
    • Prompt Strategy: "Few-Shot Prompting" (Give it 3 examples of good cold DMs).
  5. Storage:
    • Database: SQLite (Local file) for session management (to avoid scraping the same business twice).
    • Output: Pandas to export to CSV/Excel.

4. Data Logic & Validation Rules

Module A: The Maps Scraper

  • De-Duplication: Check SQLite database. If Business Name + Zip Code exists, SKIP.
  • Filter: Skip businesses with < 5 Reviews (Too new/risky) or > 500 Reviews (Too big/corporate).

Module B: The Crawler

  • Email Priority:
    1. Look for specific names (rahul@, ceo@).
    2. Look for functional (info@, contact@).
    3. Ignore trash (noreply@, sentry@).
  • Context Hook: Grab the first <h1> tag or the first <date> tag on the blog page.

Module C: The Writer

  • Constraint: Max tokens = 150.
  • Tone Check: If the generated text contains "I hope this email finds you well," Regenerate (Strict penalty).

6. Risks & Mitigation Strategies

  1. Google Maps Ban (IP Blocking):
    • Risk: Google detects the bot and serves a CAPTCHA.
    • Mitigation: Use stealth_playwright. Implement random sleep(2, 10) seconds between scrapes. Limit to 50 leads/run.
  2. Bad Data ( hallucinations):
    • Risk: AI writes "I loved your article on X" when X doesn't exist.
    • Mitigation: Human Review Step. The system outputs a draft, not a sent message. The human must hit "Send."
  3. Spam Trap Emails:
    • Risk: Sending to a "honeypot" email ruins domain reputation.
    • Mitigation: Integrate a free Python library like email-validator to ping the SMTP server before saving the lead.

7. Next Steps for Architecture Selection

Since you are selecting components, I recommend:

  1. Test the Scraper: Run a simple Playwright script on your local machine to see if Google Maps blocks you.
  2. Test the Model: Get a free API key from Groq or DeepSeek and test the prompt quality manually.
